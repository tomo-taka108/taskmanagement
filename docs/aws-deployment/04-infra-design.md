# 04. インフラ設計 — 使用するAWSサービスの役割解説

`00-overview.md`のアーキテクチャ図に登場した各サービスについて、1つずつ「何のためにあるか」を解説します。

---

## 1. VPC（Virtual Private Cloud）

VPCは、AWSアカウントの中に作る**自分専用の仮想ネットワーク**です。
「10.0.0.0/16」のようなIPアドレス範囲を指定して作成します。

家に例えると、VPCは「敷地」です。この敷地の中にどんな部屋（サブネット）を作るか、
どこに窓や玄関（インターネットへの出入り口）をつけるかを、この後決めていきます。

今回は`10.0.0.0/16`（約6万5千個のIPアドレスが使える範囲）を1つ作ります。

## 2. サブネット（Subnet）とパブリック/プライベートの違い

サブネットはVPCの中をさらに区切った小さな区画です。今回は「パブリック」と「プライベート」を2種類ずつ（異なるAZ = アベイラビリティゾーンに1つずつ、合計4つ）作ります。

- **パブリックサブネット**: インターネットへの出入り口（インターネットゲートウェイ）に繋がっているサブネット。ALBやFargateタスクなど、外部からアクセスされる／外部にアクセスする必要があるものを置く
- **プライベートサブネット**: インターネットへの出入り口を持たないサブネット。RDS（データベース）を置く

なぜAZ（アベイラビリティゾーン、データセンターの単位）を分けて2つずつ作るかというと、
RDSやALBが「最低2つの異なるAZにまたがったサブネットグループ」を要求する仕様になっているためです
（可用性を高めるための設計ですが、今回は動作要件を満たすために最小限2つ用意する、という理解で十分です）。

## 3. インターネットゲートウェイ（IGW）とNAT Gatewayについて

- **インターネットゲートウェイ**: VPCとインターネットを繋ぐ「玄関」。パブリックサブネットのルートテーブルに「0.0.0.0/0 (全部) はIGW経由」という設定をすることで、そのサブネット内のリソースがインターネットと通信できるようになります
- **NAT Gateway**: 本来は「プライベートサブネットにあるリソースが、外部からの着信は受けずに、外部への発信だけできるようにする」ための仕組みです。多くのAWS構成ガイドではRDSやFargateをプライベートサブネットに置き、NAT Gateway経由で外部（パッケージ取得など）にアクセスさせます

**今回はNAT Gatewayを使いません。** 理由はコストです。NAT Gatewayは起動しているだけで月$30以上かかり、
学習用途のコストとしては大きすぎます。代わりに以下の設計にしています。

- **Fargateタスク（バックエンド）はパブリックサブネットに配置**し、パブリックIPを直接割り当てます。ただし後述のセキュリティグループで「ALBからの通信だけ受け付ける」ように制限するため、外部から直接攻撃されるリスクは抑えられています
- **RDS（データベース）はプライベートサブネットに配置**し、外部インターネットとの通信経路自体を持たせません。ECSのセキュリティグループからの通信のみ許可します（RDSは基本的に外向き通信が不要なので、NATがなくても問題なく動作します）

## 4. セキュリティグループ（SG）

セキュリティグループは、リソースごとに設定する「ファイアウォール」です。
「どこから来た、どのポート宛の通信を許可するか」を定義します。今回は3つ作ります。

| SG | 用途 | 許可するルール |
|---|---|---|
| ALB用SG | ALBにアタッチ | 誰からでも80番ポート（HTTP）を許可 |
| ECS用SG | Fargateタスクにアタッチ | ALB用SGからの8080番ポートのみ許可 |
| RDS用SG | RDSにアタッチ | ECS用SGからの5432番ポート（PostgreSQL）のみ許可 |

この「SGからSGへの許可」という連鎖により、「インターネット → ALB → ECS → RDS」という一方向の経路だけが開き、
RDSに直接インターネットからアクセスすることは（IPが分かったとしても）できない設計になっています。

## 5. ALB（Application Load Balancer）

ALBは、通信を受け取って適切な宛先（ここではECSタスク）に振り分ける「受付」です。
今回はFargateタスクが1つしかないため「振り分け」の恩恵はほぼありませんが、
ECS Fargateサービスの標準的な構成として、ALBを経由してヘルスチェック（正常に動いているかの確認）を行う役割も持ちます。

学習目的のため、HTTPS化（証明書が必要）は行わず、HTTP(80番)のみで構成します。

## 6. ECS（Elastic Container Service）と Fargate

ECSはDockerコンテナを管理・実行するAWSのサービスです。「タスク定義」（どのイメージを、どれだけのCPU/メモリで動かすか）と「サービス」（タスクを何個起動し続けるか、ALBとどう連携するか）を組み合わせて動きます。

Fargateは、ECSの「起動タイプ」の1つで、**裏側のサーバー（EC2インスタンス）を意識しなくてよい**サーバーレスな実行環境です。
「EC2起動タイプ」だとサーバーのOSアップデートなどを自分で管理する必要がありますが、Fargateはそれが不要です。

さらに今回は**Fargate Spot**という起動方式を使います。これは「AWS側の余剰キャパシティを間借りする」形の割引プランで、
通常のFargate料金より約70%安くなります。デメリットは「AWSの都合で稀に強制的にタスクが停止させられることがある」点ですが、
学習用の1タスク構成であれば許容範囲と判断しています。

## 7. ECR（Elastic Container Registry）

ECRは、Dockerイメージを保管しておく「倉庫」です。GitHubがコードを保管する場所なら、ECRはDockerイメージを保管する場所、とイメージすると分かりやすいです。

流れとしては、「バックエンドのコードをDockerイメージとしてビルド → ECRにpush（アップロード） → ECS Fargateがそのイメージをpull（ダウンロード）して実行」となります。

## 8. RDS（Relational Database Service）

RDSは、PostgreSQLやMySQLなどのデータベースを、AWSが構築・運用・バックアップまで面倒を見てくれるマネージドサービスです。
自分でEC2にPostgreSQLをインストールして運用する場合と違い、OSパッチ適用やバックアップ設定などの手間が省けます。

今回は`db.t4g.micro`という最小クラスのインスタンスタイプを使います。これはAWSの無料利用枠（新規アカウントから12ヶ月）の対象になりやすいスペックです。

## 9. S3（Simple Storage Service）

S3はファイルを保存しておくオブジェクトストレージです。今回はReactのビルド成果物（`npm run build`で生成される`dist/`フォルダの中身、HTML/CSS/JSファイル）をアップロードする先として使います。

S3単体でも「静的ウェブサイトホスティング」機能がありますが、今回はCloudFrontと組み合わせるため、
S3バケット自体は非公開（パブリックアクセスブロック有効）にし、CloudFront経由でのみアクセスできるようにします。

## 10. CloudFront

CloudFrontはCDN（Content Delivery Network）です。世界中に配置されたAWSのエッジサーバーが、S3にあるファイルをキャッシュして配信することで、ユーザーに近い場所から高速に配信します。

今回CloudFrontを使うもう1つの理由は**HTTPS対応**です。CloudFrontはデフォルトで発行される`*.cloudfront.net`というドメインに対して、
追加費用なしで自動的にHTTPS証明書を提供してくれます。独自ドメインを取得していない今回の学習環境でも、フロントエンドを安全な接続で配信できます。

CloudFrontとS3の接続には**OAC（Origin Access Control）**という仕組みを使います。これは「CloudFrontからのアクセスだけをS3に許可し、
S3バケットへの直接アクセスは拒否する」ための設定です。

また、ReactのようなSPA（Single Page Application）では、`/board`のようなURLに直接アクセスすると本来S3には対応するファイルが存在せず404になってしまうため、
「404/403が返ってきたら`index.html`を200で返す」というカスタムエラーレスポンス設定をCloudFrontに入れます。

## 11. IAM Identity Center（再掲）

`01-aws-account-setup.md`で設定済みですが、改めて位置づけを説明すると、IAM Identity Centerは「誰が」「どのAWSアカウントに対して」「どんな権限で」ログインできるかを管理する仕組みです。
長期的な「アクセスキー」を発行して使い回す従来方式に比べ、一時的な認証情報がブラウザ認証のたびに発行される点で安全性が高く、AWSも現在この方式を推奨しています。
