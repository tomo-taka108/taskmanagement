# 04. インフラ設計 — 使用するAWSサービスの役割解説

`00-overview.md`のアーキテクチャ図に登場した各サービスについて、1つずつ「何のためにあるか」を解説します。

---

## 1. VPC（Virtual Private Cloud）

VPCは、AWSアカウントの中に作る**自分専用の仮想ネットワーク**です。
「10.0.0.0/16」のようなIPアドレス範囲を指定して作成します。

家に例えると、VPCは「敷地」です。この敷地の中にどんな部屋（サブネット）を作るか、
どこに窓や玄関（インターネットへの出入り口）をつけるかを、この後決めていきます。

今回は`10.0.0.0/16`（約6万5千個のIPアドレスが使える範囲）を1つ作ります。

## 2. サブネット（Subnet）とパブリック/プライベートの違い

サブネットはVPCの中をさらに区切った小さな区画です。今回は「パブリック」と「プライベート」を2種類ずつ（異なるAZ = アベイラビリティゾーンに1つずつ、合計4つ）作ります。

- **パブリックサブネット**: インターネットへの出入り口（インターネットゲートウェイ）に繋がっているサブネット。EC2インスタンスなど、外部からアクセスされるものを置く
- **プライベートサブネット**: インターネットへの出入り口を持たないサブネット。RDS（データベース）を置く

なぜAZ（アベイラビリティゾーン、データセンターの単位）を分けて2つずつ作るかというと、
RDSが「最低2つの異なるAZにまたがったサブネットグループ」を要求する仕様になっているためです
（可用性を高めるための設計ですが、今回は動作要件を満たすために最小限2つ用意する、という理解で十分です）。
EC2は1台だけなので本来AZを分ける必要はありませんが、VPCの構成をシンプルに保つため、パブリックサブネットも同様に2つ作ります。

## 3. インターネットゲートウェイ（IGW）とNAT Gatewayについて

- **インターネットゲートウェイ**: VPCとインターネットを繋ぐ「玄関」。パブリックサブネットのルートテーブルに「0.0.0.0/0 (全部) はIGW経由」という設定をすることで、そのサブネット内のリソースがインターネットと通信できるようになります
- **NAT Gateway**: 本来は「プライベートサブネットにあるリソースが、外部からの着信は受けずに、外部への発信だけできるようにする」ための仕組みです。多くのAWS構成ガイドではRDSをプライベートサブネットに置き、NAT Gateway経由で外部（パッチ適用など）にアクセスさせます

**今回はNAT Gatewayを使いません。** 理由はコストです。NAT Gatewayは起動しているだけで月$30以上かかり、
個人利用のコストとしては大きすぎます。代わりに以下の設計にしています。

- **EC2インスタンス（フロント+バックエンド）はパブリックサブネットに配置**し、パブリックIPを直接割り当てます。セキュリティグループで「80番ポート（HTTP）を全世界に許可、それ以外は自分のIPからのSSHのみ許可」のように必要最小限に制限します
- **RDS（データベース）はプライベートサブネットに配置**し、外部インターネットとの通信経路自体を持たせません。EC2用セキュリティグループからの通信のみ許可します（RDSは基本的に外向き通信が不要なので、NATがなくても問題なく動作します）

## 4. セキュリティグループ（SG）

セキュリティグループは、リソースごとに設定する「ファイアウォール」です。
「どこから来た、どのポート宛の通信を許可するか」を定義します。今回は2つ作ります。

| SG | 用途 | 許可するルール |
|---|---|---|
| EC2用SG | EC2インスタンスにアタッチ | 誰からでも80番ポート（HTTP）を許可。自分のIPからのみ22番ポート（SSH）を許可 |
| RDS用SG | RDSにアタッチ | EC2用SGからの5432番ポート（PostgreSQL）のみ許可 |

この「SGからSGへの許可」により、「インターネット → EC2 → RDS」という一方向の経路だけが開き、
RDSに直接インターネットからアクセスすることは（IPが分かったとしても）できない設計になっています。

## 5. EC2（Elastic Compute Cloud）

EC2は、仮想サーバーを1台単位で借りられるAWSの基本的なサービスです。
今回はこのEC2インスタンス1台の上で、Docker Composeを使って以下の2つのコンテナを動かします。

- **Nginxコンテナ**: Reactのビルド成果物（静的ファイル）を配信しつつ、`/api`宛のリクエストをバックエンドコンテナにリバースプロキシする
- **Spring Bootコンテナ**: バックエンドAPI。8080番ポートでNginxからのリクエストのみを受け付ける（外部に直接ポートは公開しない）

インスタンスタイプは`t3.micro`を使います。これはAWSの無料利用枠（新規アカウントから12ヶ月間、750時間/月）の対象です。
ECS FargateやALBのような「サーバーレスで自動スケール」の仕組みはありませんが、個人利用で同時アクセスが少ない前提であれば、
EC2 1台で十分に動作します。

構成がシンプルな分、学習の見通しも良くなります。「サーバーを1台借りて、その中でDockerコンテナを動かす」という
最もオーソドックスな形なので、AWS特有の複雑な概念（ECSのタスク定義・サービス、ALBのターゲットグループなど）を
理解しなくても構築できます。

## 6. RDS（Relational Database Service）

RDSは、PostgreSQLやMySQLなどのデータベースを、AWSが構築・運用・バックアップまで面倒を見てくれるマネージドサービスです。
自分でEC2にPostgreSQLをインストールして運用する場合と違い、OSパッチ適用やバックアップ設定などの手間が省けます。

今回は`db.t4g.micro`という最小クラスのインスタンスタイプを使います。これはAWSの無料利用枠（新規アカウントから12ヶ月）の対象になりやすいスペックです。

（アプリケーションと同じEC2インスタンス内のDockerコンテナとしてPostgreSQLを動かす選択肢もありますが、
今回はRDSのバックアップ・運用面のメリットを活かすため、DBだけはRDSを使う構成にしています。）

## 7. IAM Identity Center（再掲）

`01-aws-account-setup.md`で設定済みですが、改めて位置づけを説明すると、IAM Identity Centerは「誰が」「どのAWSアカウントに対して」「どんな権限で」ログインできるかを管理する仕組みです。
長期的な「アクセスキー」を発行して使い回す従来方式に比べ、一時的な認証情報がブラウザ認証のたびに発行される点で安全性が高く、AWSも現在この方式を推奨しています。
